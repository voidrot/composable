# Copilot Agent Instructions

This file provides instructions for the GitHub Copilot coding agent when working on this repository.

## Model Selection

Use the **least powerful model** that can complete the task correctly. Default to `o4-mini` for routine tasks.

| Task type | Recommended model |
|---|---|
| Version bumps, tag updates, doc updates | `o4-mini` (default) |
| Adding new fragments or stacks | `o4-mini` |
| Refactoring CLI logic or schema changes | `claude-sonnet-4-5` |
| Complex architectural changes, multi-service rewrites | `claude-opus-4-5` |

## Project Overview

**Composable** is a Docker Compose fragment library and CLI. Fragments live in `fragments/compose/` as `.yml` + `.json` pairs. Stacks live in `stacks/` as `.json` files. Documentation is auto-generated by `scripts/generate-docs.js`.

## Available MCP Servers

The following MCP servers are configured for this repository. **Prefer these over web search** for all version and documentation lookups:

### Docker Hub MCP (`docker-hub`)
Use to look up current image tags for Docker Hub images.

Key tools:
- `list_repository_tags` — list available tags for an image (e.g. `{ namespace: "library", repository: "postgres" }`)
- `check_repository_tag` — verify a specific tag exists
- `get_repository_info` — get metadata about a repository
- `search` — search Docker Hub for images

### Context7 MCP (`context7`)
Use to look up up-to-date documentation for libraries and services.

Key tools:
- `resolve-library-id` — find the Context7 ID for a library by name
- `query-docs` — query documentation for a library by ID

## Common Tasks

### Updating Fragment Image Versions

1. **Do NOT use web search.** Use the Docker Hub MCP `list_repository_tags` tool instead.
2. Scan `fragments/compose/*.json` for version variables (e.g. `POSTGRES_VERSION`, `REDIS_TAG`).
3. For each image, call `list_repository_tags` and find the latest **stable** tag (avoid `latest`, `edge`, `nightly`, `beta`, `rc`).
4. Prefer **specific version tags** (e.g. `16.3`, `7.2.5`) over generic aliases.
5. Update the version in the `.json` `variables` field.
6. Update any hardcoded tags in the corresponding `.yml` file.
7. Run `npm run validate` to ensure schemas are satisfied.
8. Update frontmatter in any corresponding `docs/fragments/<name>.md` if it has a version field.

### Adding a New Fragment

1. Create `fragments/compose/<name>.yml` with the service definition.
2. Create `fragments/compose/<name>.json` with metadata matching the schema in `schemas/`.
3. Run `npm run validate` to verify both files pass schema validation.
4. Run `node scripts/generate-docs.js` to regenerate docs and update `mkdocs.yml`.

### Adding a New Stack

1. Create `stacks/<name>.json` following the schema in `schemas/`.
2. Run `npm run validate`.
3. Run `node scripts/generate-docs.js`.

## Validation

Always run before submitting a PR:

```bash
npm run validate
```

## Code Style

- Use YAML anchors sparingly; prefer explicit configuration in fragments.
- Keep environment variable names in `SCREAMING_SNAKE_CASE`.
- Fragment `.yml` files should define a single top-level service key.
- All required schema fields must be present in `.json` metadata files.
